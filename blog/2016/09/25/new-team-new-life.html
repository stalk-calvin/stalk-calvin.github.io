Joining a New team at the Company

I have been extremely busy these days. Normally I have enough time to do other things but these days seems to be nothing but work+gym+sleep plus a little bit of studying in between. I am trying to find a time to hang out with my wife more, but it's been mostly on Friday or Saturday for an actual date. It has been a bit more tough than before to go out on weekdays. I barely make it home by 8-9 PM after my gym session. Well, all these busy-ness comes from relocating into a new team starting this month. Honestly I have been doing enough (most of) Java with my previous role and I find the guys here are doing mostly similar things as I have been doing. However, the size of the team is about twice (if not three times) bigger than the size compare to the team I was in before. There are more developers reviewing the my changes and obviously there are more for me to spend time reviewing other people's code as well at the same time.

Full time Java Developer

On my previous role, I would say about 70-80% of time I have been concentrating writing/maintaining the code in Perl. Everyone had mutual understanding that I am hired as a Perl developer that I do not require to write in Java which motivated other Java developer to implement. No one expected us Perl developers to implement/fix in Java hence work on Java was not too much. However, now with this new team, I am a full Java Developer. I do nothing but working in Java all day. Honestly I love both languages. Well.. to be really honest, I actually find it easier to work in Java than Perl. It's probably because I like the abstraction / bounded aspects (more restrictions) Java language features. With Perl, I had many ways to solve problems and the language itself is much flexible. It could be one line of code or 10 lines of codes. The Clean Code portion was not big of a deal in Perl. One liner that other developers did not understand wasn't a big deal. It would be up to other developer to understand what was written. With Java, there are rules and people follow the standards which makes the code more readable (which is super important) but what's really important? Writing and Following these rules greatly help the future development (less errors likely pop up with clean code). I guess I could have forced Perl developers to write cleanest code as possible, but then they would argue, performance is terrible if I started doing that! Well, to me, performance will follow with you as you write code cleanly. Anyways… too much talk on clean code LOL

Conclusion

It has not even been a month since I moved to the new team to start working as a Full Jave Developer. I still have a lot more to find out about the product at the new team. I really feel like this is a good challenge and truly am excited! I should be doing this more often, switching teams I mean.. hehe. I was actually feeling like there used to be this smooth wheels rotating the same every day. However, I feel I just replaced that wheel to something really rough and I am smoothing it out! Perhaps when the wheel is all smoothed again, I will try and find another team :)
