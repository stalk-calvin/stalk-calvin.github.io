Web Security

Security of web- sites, applications and services


02 Jul 2016 Comments
#websecurity #computer

For being in a company where security is the most important piece of work: maintained and truly valued in my company. I naturally gained knowledge in most of security surrounding web development, not only to follow Owasp Top10, but also things that are not list in top 10 list. Anyone working in a secure web application environment should know owasp top 10 list. Anyway I will include some of the items that just come to my mind in the world of web security. Note below content is not everything, they are just reference of some of security that we need to enforce in areas of web technology which i find important:

XSS (Cross-site scripting)

This attack is similar to SQL injection described above, except this injection is happening with a browser which result in executing unexpected behavior such as publishing some of the user’s sensitive information, such as cookies. Keylogging is another problem, where attacker can execute a piece of perlscript command easily from either comment section or attacker can even send you a piece of link via email and record to get your sensitive information such as passwords and credit card numbers. These are some examples.

Hackers will find most creative and easy method to hack into your information or other needs. So how do we prevent all these? The resposibility to create a secure web application is really who is implementing/serving the website. The one thing if they are really afraid of this attack can disable the entire scripts from browser, but who wants to do that? Job of the server is quite easy. Make sure all characters are escaped properly. Note this does not prevent 100% either, but just a baseline of hacking method is explained here.

Encrypted Password

This is really a common sense but I thought I place it here as this is an important to note. First, you have to understand to scope of encryption. So if your website requires authentication then it is probably good idea to wrap your website with SSL protection to protect the data being transmitted between client and server. With SSL it gives minimum protection that when data is hijacked in middle, unless you have the key, the hacker will have very hard time to decrypt the context.

However, this decryption is possible (either with the actual key or just simply breaking the algorithm). So we need to make extra layer of security for data that we would like to make it secure. The example would be credit card. If we somehow compromised the data storage (db, nosql, etc), we do not want the hackers visibly see plain text of credit card #. Also, this number is something that even server owners should not see. So we can make an encryption on the card # by giving each different # a key associated where we can decrypt to either make a transaction or something when it is absolutely needed.

For passwords, which typically is encrypted as well. There are many websites that does just a basic md5/sha1 hash applied to it. This is called “hashing”, not encrypted password. This could be a good enough usage for password since we only need one way hashing. Remember to add the SALT into the hash. If you do not add SALT to this string, hackers can use dictionary attach to get the characters that matches the hash value, but this is not gonna be possible with SALT added to the string.

HttpOnly and Secure

A secure cookie can only be transmitted over an encrypted connection (i.e. HTTPS over SSL). Wont be transmitted over non-encrypted connections. This means if cookie got stolen, they wont be able to easily decrypt the content. Another security measure we can impose on HTTP header is a “HttpOnly” cookie. These HTTP defined headered cookies cannot be accessed by client-side APIs, such as JavaScript. This limits ability for hackers to various XSS activity against you. However, they are still vulnerable to XST or CSRF.

X-Frame-Options, Strict-Transport-Security and Public-Key-Pins

These are the HTTP headers to be added as part of HTTP Response to protect client from attackers. There are few more that does extra protections. I picked these 3 since I consider them the most important out of them. X-Frame-Options is a header to specify browser to not render page within a frame. There are options to choose from where you can deny displaying this content in a frame, you can allow content in a frame for same origin. Last option is all but listed allowed from domain is denyed. This header is explicitly for protecting the clients against clickjacking. Clickjacking is basically using the frame manipulating the clients to react on some actions such as voting/posts/comments/follow etc. OK, let’s talk about HTTP Strict-Transport-Security. This is the header of HTTP response saying only process this response in secure HTTPS connections. It is communicated via HTTP response header via an HTTP header field named “Strict-Transport-Security”. Lastly, for the HTTP response header protection, I’d like to talk about HTTP Public-Key-Pins. This is basically pinning the SSL certiicate so attackers cannot replace the certificate with fraudulent certificates. HPKP contains a value which is basically a SHA-256 hash of the actual certificate which then browser will verify whether the certificate is valid.

Input Validation and Error Handling

This item is not directly related to Web Security, but it is important to mention as part of it since it prevents a lot of attack measures from doing so. Input validation is nothing but validate inputs as they come in. Data validation checks that data are valid, reasonable, and secure before they are processed. For example, it could be as simple as checking for inputs of allowed characters, checking for limits, allowed range, etc. This ultimately will protect attackers from entering malformed data into input box so overall it does the prevention. When you have a text field that you can enter in only digits, it is not possible to enter in any piece of javascript codes or sql injection. Anyway, it is very important to do the input validation as it protects and filters out any invalid inputs.

Insecurity around sensitive data

Securing data like SIN #, credit card #, password etc need extra layer of security. We never want to store them as plain text. We need to apply strong security encryption surrounding it. Think about when a hacker stole the whole entire machine, now he has all of those sensitive data himself. However, with the right encryption, he will not be able to decrypt them into a plain text. So data is not entirely stolen at that point. This could also be achieved by hijacking a packet between http protocol. Even if hackers gotten these information, they won’t be easily be able to turn them back to plain text. So those extra layer of security is always required for those sensitive data.

CSRF (Cross Site Reference Forgery)

I have put this here because our company recently patched a protection around this. An example of CSRF can be when an attacker sends a link that contains something like changing a password of the “user” or modifying someone’s status, if I had enough privilege and I have already logged into the website. By clicking the link, I would automatically update the password of the “user” attacker intends to change the password.To fix this, we ended up implementing a md5 hash string tied to a session so if an attacker sends a link and it does not have the same session as I logged in, it would give a warning page to indicate me to make sure I know what I am doing.

Transport Layer Security

Well, obviously everyone knows what https (ssl) is. It is basically providing a layer for privacy and data integrity between two communicating applications. SSLs (v1, v2 and v3) have been found vulnerable (Read Here if interested) and it is recommended to use anything greater than TLS1.1. Modern browsers support at least TLS 1.0 as TLS1.0 is still widely used as the ‘best’ protocol by a lot of browsers that are not patched to the latest version. It suffers from CBC Chaining attacks and Padding Oracle attacks. PCI DSS 3.1 prohibits use of TLS 1.0 after June 30, 2016. What’s important here is to not use SSLv2, SSLv3. SSLv1 was never publicly made available.
